# dicodePing v1.6.0-rc.4

این نسخه کاندید انتشار چهارم از شاخه ۱.۶ است و **اسکنر کاملاً خودکار**،
دکمه حجم فقط-آیکون، اصلاح دریافت حجم واقعی، پینگ/حجم بر اساس ساب فعال،
و کش ۲۰ دقیقه‌ای برای splash رو ارائه می‌ده.

## 🤖 اسکنر کاملاً خودکار

قبلاً کاربر باید اول به یک سرور وصل می‌شد بعد اسکن رو بزنه. حالا:

- **یه دکمه «شروع اسکن»** — تمام.
- قبل از دکمه، یه **پیش‌نمایش مراحل** نشون داده می‌شه:
  1. اتصال خودکار به بهترین سرور منبع اصلی
  2. دریافت کانفیگ از کانال‌های تلگرام
  3. قطع اتصال و تست هم‌زمان سرورها
  4. ذخیره سرورهای سالم به‌عنوان ساب جدید
- یه hint آبی هم نوشته شده: «کافیه یک‌بار دکمه شروع اسکن رو بزنی. بقیه‌اش با خودمه!»
- بعد از زدن دکمه، برنامه خودش به بهترین سرور وصل می‌شه، کانال‌ها رو می‌گیره،
  قطع می‌کنه، تست می‌کنه و ذخیره می‌کنه. کاربر فقط نتیجه رو می‌بینه.

## 🎨 دکمه حجم فقط-آیکون

- دکمه «دریافت حجم‌ها» در toolbar صفحه سرورها حالا **فقط آیکون** هست
  (۴۴dp عرض، بدون متن).
- این باعث می‌شه layout با فونت‌های بزرگ‌تر به هم نریزه و responsive
  بمونه.
- tooltip هنوز متن «دریافت حجم‌ها» رو نشون می‌ده.

## 🔧 اصلاح دریافت حجم واقعی

- قبلاً فقط HEAD request می‌فرستادیم. برخی پروایدرها HEAD رو reject
  می‌کنن و هدر `Subscription-Userinfo` رو برنمی‌گردونن.
- حالا **اول HEAD، اگه fail شد GET با `Range: bytes=0-0`** (یه بایت
  دانلود می‌کنه تا هدر رو بگیره).
- این باعث می‌شه حجم واقعی برای خیلی بیشتر از پروایدرها کار کنه.

## 🎯 پینگ/حجم بر اساس ساب فعال

- قبلاً وقتی روی یه ساب خاص (نه «همه») بودید و دکمه پینگ یا حجم رو
  می‌زدید، کل سرورها رو پینگ/حجم می‌گرفت.
- حالا **فقط سرورهای همون ساب فعال** رو پینگ/حجم می‌گیره.
- این یعنی اگه روی یه ساب با ۲۰ تا سرور هستید، به جای ۲۰۰ تا سرور،
  فقط ۲۰ تا رو تست می‌کنه — خیلی سریع‌تر.
- در اندروید هم همینطور: دکمه refresh و pingAll وقتی روی ساب خاصی هستید،
  فقط همون ساب رو می‌گیرن.

## 💾 کش ۲۰ دقیقه‌ای برای splash

- یه ماژول cache جدید (`dicodeping/ping_cache.py`) ساخته شد که
  پینگ/لوکیشن هر سرور رو برای **۲۰ دقیقه** ذخیره می‌کنه.
- وقتی برنامه دوباره باز می‌شه، splash screen از این cache استفاده می‌کنه:
  - سرورهایی که cache شون تازه هست → از همون مقادیر قبلی استفاده می‌شه.
  - فقط سرورهای جدید یا stale رو دوباره پینگ می‌گیره.
- این یعنی اگه ۲۰ دقیقه پیش برنامه رو بستید و الان باز می‌کنید، splash
  تقریباً فوراً تموم می‌شه چون پینگ‌ها از cache میان.
- cache در `DATA_DIR/ping_cache.json` ذخیره می‌شه و بین restart ها باقی
  می‌مونه.
- `service.refresh_saved_with_cache` مسیر اصلی هست که از cache استفاده
  می‌کنه.

## 📦 هر سه پلتفرم

تمام این ویژگی‌ها به‌صورت یکسان در ویندوز، لینوکس و اندروید پیاده‌سازی
شدن:

- **دسکتاپ**: `dicodeping/ping_cache.py`، `dicodeping/service.py`
  (refresh_saved_with_cache, refresh_subset)، `dicodeping/workers.py`
  (RefreshSubsetThread)، `dicodeping/volume.py` (HEAD→GET fallback)،
  `dicodeping/ui.py` (preview stages، icon-only button، source-scoped).
- **اندروید**: `AppRepository.kt` (pingSource)، `ServersFragment.kt`
  (source-scoped refresh/ping/volume)، `fragment_servers.xml` (icon-only
  button)، `fragment_scanner.xml` (preview stages).

## 🔧 جزئیات فنی

- **ping_cache**: `CachedPing` dataclass، `apply_cached_to_records` که
  records رو به `(cached, fresh)` split می‌کنه، `update_cache` که بعد از
  هر ping واقعی صدا زده می‌شه.
- **service**: `refresh_saved_with_cache` فقط fresh subset رو پینگ می‌گیره
  و cached رو as-is برمی‌گردونه. `refresh_subset` فقط IDs مشخص رو
  پینگ می‌گیره.
- **volume**: `_try` helper که HEAD و GET رو امتحان می‌کنه.
- **آزمون‌ها**: `tests/test_v160_rc4.py` با ۱۰ تست جدید. کل ۱۱۶ تست pass شد.
